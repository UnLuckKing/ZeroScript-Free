# ZeroScript One 1.36 — Selected Upstream Reliability Sync

Upstream `sebattfg/ZeroScript-Free` releases 1.4.1–1.4.3 were reviewed against the simplified ZeroScript One architecture. The release ports the reliability improvements that reduce waiting and connection failures without restoring provider clutter.

## Added

- TCP and TCPv6 ownership detection for Roblox Studio's MCP port 13469.
- PID/process-tree detection for a leftover `StudioMCP.exe` that belongs to an older bridge session.
- Recovery for a proven non-Roblox WebSocket host/port squatter.
- Clear command-not-found messages for optional MCP server configuration mistakes.
- Qwen stable response identity using the `chat-response-message-<uuid>` descendant.
- Qwen outbound-result cap with a marked head/tail-preserving truncation.
- Refusal-resistant bootstrap wording that explains why ZeroScript text commands are the channel that reaches local Studio.

## Changed

- The one-click hidden bridge never waits on an invisible `input()` prompt. Known squatters are handled automatically; unknown conflicts are logged clearly without freezing startup.
- After Studio MCP recovery, the bridge tells the user to toggle **Assistant Settings > MCP Servers > Enable Studio as MCP server** so Studio performs its one-shot registration again.

## Intentionally excluded

- Meta AI provider from upstream 1.4.3. ZeroScript One remains ChatGPT-first, with existing providers serving only as fallbacks.
- Extra support/donation menu changes, which do not improve Roblox game creation speed or reliability.
- A blind full upstream merge, which would overwrite ZeroScript One's prototype, Memory Vault, desktop workspace and ChatGPT Max layers.

## Validation

- Added Python tests for IPv4/IPv6 Studio port parsing.
- Added a JavaScript release-sync test covering manifest loading, prompt routing and Qwen fixes.
- Updated extension, bridge, control API, desktop launcher, build validator and CI version parity to 1.36.0.

Live Windows, Chrome and Roblox Studio verification is still required because process ownership and provider DOM behavior cannot be reproduced in repository-only tests.
